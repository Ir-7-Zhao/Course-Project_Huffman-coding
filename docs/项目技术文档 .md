# Project：Huffman编码
## Huffman编码原理讲解
### 引入：香农信息熵
大一上计算机科学导论课程中提及的**香农信息熵**，是衡量信息不确定性的核心指标，也是无损数据压缩的理论极限。信息熵代表了**编码一段数据所需的最小平均比特数**，任何无损压缩算法都无法超越这个极限。

#### 信息熵计算公式
对于一个离散随机变量 X，其可能取值为 x1, x2, ..., xn，每个取值出现的概率为 p(xi)，则**信息熵 H(X)** 计算公式为：
$$H(X) = -\sum_{i=1}^{n} p(x_i) \log_2 p(x_i)$$

- 单位：比特(bit)
- 物理意义：数据的理论最小平均编码长度，哈夫曼编码的目标就是让实际平均编码长度无限逼近这个值。
- 举例：若字符a概率50%、b25%、c25%，熵 $H = -0.5log_20.5 -0.25log_20.25 -0.25log_20.25 = 1.5 bit$，哈夫曼编码可恰好达到该理论值。

哈夫曼编码作为经典的无损压缩算法，其设计本质是让出现频率高的字符使用更短的二进制编码，频率低的字符使用较长编码，从而最小化整体编码长度，严格逼近香农信息熵的理论下限。

### 香农-范诺编码

香农-范诺编码是网络信息传输场景下的经典编码方式，核心思路是**自顶向下**对字符频率进行近似**二分划分**：

先将字符按频率从高到低排序，再将排序后的字符集合拆分为两个子集合，使两个子集合的总频率尽可能接近，随后对两个子集合分别分配0和1编码，重复此过程直至每个字符都有唯一编码。该编码方式实现简单，但由于是“**近似二分**”，无法保证编码长度严格最优。

### 哈夫曼编码
哈夫曼编码弥补了香农-范诺编码的不足，采用**自底向上**的构建方式，**严格按照字符出现概率（频率）大小进行精准编码**。其核心逻辑为：
1. 统计每个字符的出现频率；
2. 以每个字符为叶子节点，构建小根堆（优先队列），堆中节点按频率升序排列；
3. 反复取出堆中频率最小的两个节点，合并为一个新的中间节点（频率为两节点之和），并将新节点放回堆中；
4. 直至堆中仅剩一个节点，该节点即为哈夫曼树的根节点；
5. 从根节点出发，向左遍历记为0、向右遍历记为1，最终每个叶子节点对应的0/1串即为该字符的哈夫曼编码。
哈夫曼编码能保证单字符编码的最优性，是无损压缩领域的基础算法。

## 哈夫曼算法实现代码简介

**注**： 本项目实现的哈夫曼编码算法仅在程序内存中完成字符/二进制数据的编码与解码过程，程序运行期间不会在磁盘中生成、写入或保存任何外部文件。 程序输出内容仅为算法执行过程中的中间数据展示、编码结果打印、压缩率计算等控制台信息，不产生可双击打开的压缩文件或解压文件。 本项目核心目的为演示哈夫曼编码的原理与算法实现逻辑，而非生成可存储的实体压缩文件。

### 一、整体实现思路（核心流程）
本项目采用C++ + STL容器实现哈夫曼编解码，整体遵循

**【数据读取→频率统计→构建哈夫曼树→生成编码表→编码压缩→解码还原】**

的闭环流程，所有设计都围绕算法逻辑匹配、执行效率最优、代码可读性强三个核心目标：

1. 输入层：读取文本/图片/音频的二进制原始数据，统一用字节流存储；
2. 统计层：快速统计每个字节（字符）的出现频率；
3. 核心算法层：用小根堆实现哈夫曼树的自底向上构建，递归生成编码；
4. 转换层：将二进制编码串转换为可存储的字节流（压缩），反向还原原始数据（解码）；
5. 输出层：写入压缩/解压后的文件，完成完整编解码功能。

所有数据结构、函数编写，基本贴合哈夫曼算法的数学逻辑，同时利用STL特性最大化执行效率。

### 二、代码程序中使用的STL组件
#### unordered_map（无序映射表）

类型：关联容器（哈希表实现）

算法逻辑绑定：哈夫曼编码第一步必须建立【字符→频率】【字符→编码】的一对一映射关系，这是算法的基础数据需求。

**为什么选这个容器：**

- 平均 O(1) 查找/插入效率，远高于红黑树实现的map，处理海量文件数据时速度更快；

- 键值对结构天然匹配算法的映射需求，代码逻辑与算法原理完全对齐，可读性极强；

- 支持unsigned char（字节）作为键，完美适配二进制文件处理。

**函数编写逻辑：**

- countFrequency：遍历字节流，用unordered_map累加频率，一行代码完成统计，直接对应算法【频率统计】步骤；

- buildCodeTable：存储最终编码表，解码时可通过字符瞬间查到对应二进制串。

**对应CPP代码实现：**

```cpp
  // 对应算法：频率统计 → 用unordered_map实现O(1)频率累加
  unordered_map<unsigned char, int> countFrequency(const vector<unsigned char>& data) {
    unordered_map<unsigned char, int> freq;
    for (unsigned char c : data) freq[c]++;
    return freq;
  }

// 对应算法：生成编码表 → 键值对存储字符与编码的映射
void buildCodeTable(Node* root, string code, unordered_map<unsigned char, string>& table) {
    if (!root->left && !root->right) {
        table[root->ch] = code;
        return;
    }
    buildCodeTable(root->left, code + "0", table);
    buildCodeTable(root->right, code + "1", table);
}
```

#### priority_queue（优先队列 / 小根堆）

类型：容器适配器（底层默认基于vector实现）

算法逻辑绑定：哈夫曼树构建的核心操作是【每次取出频率最小的两个节点】，这是小根堆的标准应用场景。

**为什么选这个容器：**

- 无需手动实现堆排序，STL封装好的小根堆直接对应算法【取最小节点】逻辑；
- 插入/取堆顶 O(log n) 效率，是构建哈夫曼树的最优数据结构；
- 自定义比较函数Compare，可精准控制「频率小的优先出队」，完全匹配算法规则。

**函数编写逻辑：**
- buildHuffmanTree：先将所有字符节点入堆，再循环【取两个最小节点→合并→入堆】，代码每一行都对应算法的构建步骤；
- 不使用数组/链表：数组需要反复排序，效率极低；链表无法快速取最小值，均不满足算法需求。

**对应CPP代码实现：**
```cpp
// 小根堆比较规则 → 严格对应算法：频率最小优先
struct Compare {
    bool operator()(const Node* a, const Node* b) {
        return a->freq > b->freq;
    }
};

// 对应算法：构建哈夫曼树 → 用priority_queue自动维护最小节点
Node* buildHuffmanTree(const unordered_map<unsigned char, int>& freq) {
    priority_queue<Node*, vector<Node*>, Compare> heap;
    for (auto& p : freq) heap.push(new Node(p.first, p.second));

    while (heap.size() > 1) {
        Node* left = heap.top(); heap.pop();
        Node* right = heap.top(); heap.pop();
        Node* merge = new Node(0, left->freq + right->freq);
        merge->left = left;
        merge->right = right;
        heap.push(merge);
    }
    return heap.empty() ? nullptr : heap.top();
}
```

#### vector（动态数组）

类型：序列容器

算法逻辑绑定：哈夫曼编码处理的是连续二进制数据流，需要一个能动态扩容、支持随机访问的连续内存容器。

**为什么选这个容器：**

- 连续内存布局，读取二进制文件时可直接与文件IO对接，fread直接写入vector，效率最高；
- 动态自动扩容，无需手动管理内存，避免缓冲区溢出；
- 随机访问 O(1)，编码/解码时遍历数据无性能损耗。

**函数编写逻辑：**

- readFile/writeFile：用vector<unsigned char>作为文件数据载体，直接对应算法的「原始数据/编码数据」存储；
- huffmanEncode：将二进制编码按8位打包为字节，存入vector，完成编码到可存储数据的转换，贴合压缩算法的最终输出需求。

对应CPP代码实现：
```cpp
// 对应算法：读取原始数据 → vector承载连续二进制流
vector<unsigned char> readFile(const string& filename) {
    FILE* f = fopen(filename.c_str(), "rb");
    if (!f) return {};
    fseek(f, 0, SEEK_END);
    long size = ftell(f);
    fseek(f, 0, SEEK_SET);
    vector<unsigned char> data(size);
    fread(data.data(), 1, size, f);
    fclose(f);
    return data;
}

// 对应算法：编码压缩 → vector存储打包后的字节数据
vector<unsigned char> huffmanEncode(const vector<unsigned char>& data, unordered_map<unsigned char, string>& table) {
    string bits;
    for (unsigned char c : data) bits += table[c];
    vector<unsigned char> result;
    int len = bits.size();
    for (int i = 0; i < len; i += 8) {
        string byteStr = bits.substr(i, 8);
        unsigned char byte = 0;
        for (char b : byteStr) byte = (byte << 1) | (b == '1');
        result.push_back(byte);
    }
    return result;
}
```

#### string（字符串）
类型：序列容器（字符序列）

算法逻辑绑定：哈夫曼编码是0/1二进制字符串，需要一个支持拼接、截取的文本容器。

**为什么选这个容器：**

- +运算符直接拼接0/1，代码与「左0右1」的算法规则完全一致；
- substr可快速截取8位二进制串，方便转换为字节；
- 自动管理内存，处理长编码串无风险。

**函数编写逻辑：**

- buildCodeTable：递归遍历树时，用string拼接0/1，一行代码实现编码生成，直观体现算法规则；
- 作为二进制串的临时载体，衔接「字符编码」与「字节打包」两个步骤，让流程更清晰。

**对应CPP代码实现：**

```cpp
// 对应算法：遍历哈夫曼树生成编码 → string拼接0/1
void buildCodeTable(Node* root, string code, unordered_map<unsigned char, string>& table) {
    if (!root) return;
    if (!root->left && !root->right) {
        table[root->ch] = code.empty() ? "0" : code;
        return;
    }
    buildCodeTable(root->left, code + "0", table);
    buildCodeTable(root->right, code + "1", table);
}
```

## 实际生活中Huffman编码的运用
### 以ASCII、BMP、WAV(文本、图片、音频)文件为例的Huffman编码压缩效率的演示
代码中通过模拟数据和真实文件两种方式验证了哈夫曼编码的压缩效率，不同类型数据的压缩效果差异显著：
- 文本类（ASCII）：压缩率约25%~60%。文本数据存在大量重复字符（如空格、字母、数字），频率分布不均，哈夫曼编码能有效缩短高频字符的编码长度，压缩效果显著；
- 简单图片（BMP）：压缩率约50%~80%。模拟16x16 RGB BMP图片的像素数据存在规律性重复，具备一定压缩空间，但像素值分布较文本更均匀，压缩率略低；
- 实拍图/原声（BMP/WAV）：压缩率约80%~95%。真实图片和音频数据的冗余度降低，像素/音频采样值分布更随机，哈夫曼编码的压缩效果有限；
- 压缩包/视频/随机数据：压缩率约95%以上。此类数据已经过专业压缩或本身无冗余，哈夫曼编码几乎无法压缩。

### 现代常用的ZIP压缩格式的两大基础算法
ZIP压缩格式结合了两种核心算法，实现了比纯哈夫曼编码更优的压缩效果：

1. LZ77算法：作为前置处理步骤，通过查找数据中的连续重复串，将后续重复的字符序列替换为“(偏移量, 长度)”的引用形式，大幅降低数据冗余；
2. Huffman编码：对LZ77处理后的数据（包括原始字符、偏移量、长度等）进行单字符最优无损编码，进一步压缩数据体积。
3. 两者结合既解决了哈夫曼编码对连续重复串压缩不足的问题，又保留了哈夫曼编码的最优性，是ZIP格式高效压缩的核心原因。
